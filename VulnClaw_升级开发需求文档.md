# VulnClaw 升级开发需求文档

> 更新日期：2026-05-09  
> 适用版本：当前工作区 `v0.2.7` 基线  
> 文档目标：为 VulnClaw 下一阶段升级提供可执行的产品与工程需求说明  
> 参考依据：当前仓库代码结构、现有 README 路线图、`VulnClaw_项目分析与优化报告.md`、本地测试结果 `273 passed / 3 failed / 1 skipped`

---

## 1. 文档目的

本文档用于回答两个问题：

1. VulnClaw 接下来最值得优先升级的点是什么。
2. 这些升级应如何拆分为可交付、可验收、可排期的开发需求。

本文档不是单纯的问题盘点，而是下一阶段的开发蓝图。目标是将 VulnClaw 从“功能丰富的研究型 CLI Agent”升级为“更稳定、可恢复、可验证、可观测、具备 Web 控制面的本地安全工作台”。

---

## 2. 当前现状判断

### 2.1 当前优势

VulnClaw 已经具备以下基础能力：

- Python CLI + REPL 交互入口已经成型。
- Agent 主循环、阶段推进、反死循环、CTF 模式、上下文压缩、技能分发、知识库注入都已存在。
- `target_state` 已经提供目标历史状态、快照、回滚、恢复建议等能力。
- 报告生成链路比较完整，支持会话报告、周期报告、PoC 生成。
- 新版 Web UI 已具备配置、任务、目标状态、快照、报告预览等 Phase 1 功能。
- 测试覆盖较广，核心模块已有较好的回归保护。

### 2.2 当前核心短板

结合代码与测试状态，当前升级重点不应继续优先堆新功能，而应优先补齐以下短板：

1. MCP 集成仍以 `schema 暴露 + placeholder` 为主，真实可执行能力有限。
2. `python_execute` 仅是弱限制，不是强隔离，安全边界不足。
3. 漏洞“发现 -> 验证 -> 收敛 -> 报告”的闭环还不够硬，误报/待验证结果与正式结果边界仍可加强。
4. CLI、自动循环、持久化循环、Web 任务虽然共享核心，但任务编排与状态契约尚未完全统一。
5. `target_state` 已经有价值，但仍缺少 schema 版本化、差异展示、恢复策略透明化等治理能力。
6. Web UI 当前偏“数据接线版”，离真正可操作的 Beta 控制台还有明显距离。
7. 测试虽然总体通过，但已经出现测试与实现漂移的信号，例如 `tests/test_think_filter.py` 仍引用已迁移的方法。
8. 发布工程、类型约束、跨平台 CI、构建产物一致性仍未完全收口。

### 2.3 升级判断

下一阶段建议把 README 中偏“继续横向扩功能”的路线，调整为“先稳内核，再扩能力”的策略：

- `v0.3.x` 主打稳定性、协议化、可恢复、可验证、Web Beta。
- `v0.4.x` 再推进知识库增强、语义检索、逆向/移动端能力扩展。
- `v1.0` 再做正式发布级别的工程封版。

## 3. 升级目标

### 3.1 总体目标

将 VulnClaw 升级为一个具备以下特征的本地安全工作台：

- 更稳定：网络抖动、工具异常、会话中断时能够优雅恢复。
- 更可信：漏洞结论有明确证据等级，报告默认只呈现可证明结果。
- 更安全：高风险执行能力具备明确的隔离策略、审计和模式开关。
- 更可控：CLI、Web、目标状态、报告、快照使用统一的数据契约。
- 更可观测：任务过程、工具调用、异常原因、恢复策略都可追踪。
- 更可交付：具备跨平台测试、类型检查、版本产物一致性和发布流程。

### 3.2 本阶段非目标

以下内容不建议在本阶段优先投入：

- 多租户 SaaS 化平台。
- 公网托管服务端。
- 大规模分布式任务调度。
- 在核心稳定性未完成前继续横向新增大量新 Skill。
- 在本地单机场景未打磨完成前引入复杂权限系统。

## 4. 本阶段范围

本阶段升级范围分为 7 个主题：

1. Agent 内核与任务编排统一。
2. MCP 生命周期与工具执行能力升级。
3. 高风险执行能力的安全边界增强。
4. 漏洞验证、证据分级与报告闭环升级。
5. Target State 治理与恢复能力增强。
6. Web UI 从数据接线版升级为可用 Beta 控制台。
7. 测试、类型、构建与发布工程升级。

---

## 5. 详细需求

### 5.1 模块 A：Agent 内核与任务编排统一

#### 目标

统一 CLI、自动循环、持续性循环、Web 任务的执行模型，降低状态分叉和重复逻辑。

#### 需求

1. 抽象统一的 Task Orchestrator。
   - 将 `chat`、`auto_pentest`、`persistent_pentest`、Web task 运行流程统一到同一编排层。
   - CLI 与 Web 仅作为不同的入口层，不各自维护核心执行分支。

2. 统一任务生命周期。
   - 标准状态至少包括 `pending / restoring / running / waiting_retry / completed / failed / stopped`。
   - 每个任务都要有统一的开始时间、结束时间、最后阶段、最后消息、错误原因。

3. 统一结果对象契约。
   - 对 `AgentResult`、`PersistentCycleResult`、Web `TaskEvent`、Target State Resume Meta 进行结构化收口。
   - 禁止不同入口使用同义但结构不同的字段。

4. 统一中断恢复逻辑。
   - 键盘中断、Web stop、LLM 重试耗尽、工具失败、快照恢复都要进入统一状态机。

5. 提供任务级别审计摘要。
   - 每个任务结束后自动生成一份 machine-readable summary，供报告、Web UI、target-state 复用。

#### 验收标准

- `run / recon / scan / exploit / persistent / web task` 均使用同一套编排状态模型。
- 任务中断后能够正确记录结束原因，不再出现“完成但实际中断”的状态混淆。
- Web SSE、CLI 摘要、target-state 中的任务字段含义一致。

---

### 5.2 模块 B：MCP 生命周期与工具执行能力升级

#### 目标

把 MCP 从“以 schema 为主的预览态”升级为“有明确能力分层和可执行范围的真实工具运行层”。

#### 需求

1. 建立 MCP 能力分层。
   - 支持 `local / sdk / subprocess / sse / placeholder` 五种执行模式的统一声明与展示。
   - 在 `doctor`、Web UI、日志中明确展示每个服务当前真实能力，而不是仅显示已注册。

2. 实现真实 session 生命周期管理。
   - 服务启动、握手、工具发现、健康检查、停机释放应形成闭环。
   - 对失败服务做降级而非静默失效。

3. 建立工具执行错误分类。
   - 区分配置错误、服务未启动、超时、参数错误、执行失败、协议不兼容。
   - 所有工具错误返回统一结构，而不是散落的自由文本。

4. 明确第一阶段真实接入目标。
   - P0：`fetch`、`memory`
   - P1：`chrome-devtools`、`burp`
   - P2：`adb-mcp`、`frida-mcp`、`jadx`、`ida-pro-mcp`

5. 增加工具超时、重试和熔断策略。
   - 单工具连续失败达到阈值后，在当前任务内短期禁用并给出替代建议。

#### 验收标准

- 至少 4 个 MCP 服务在非 placeholder 模式下可真实执行。
- `doctor` 和 Web UI 可准确显示服务状态、执行模式、工具数、最近错误。
- 工具失败时，Agent 能拿到结构化错误并做出降级决策。

---

### 5.3 模块 C：高风险执行能力与安全边界升级

#### 目标

让 `python_execute` 和其他高风险动作具备清晰、可配置、可审计的安全模式。

#### 需求

1. 引入执行安全级别。
   - `safe`：禁止网络和文件 IO
   - `lab`：允许目标访问，限制本地敏感路径
   - `trusted-local`：面向本地受控环境，保留现有能力

2. 强化 `python_execute` 隔离。
   - 使用独立临时工作目录。
   - 限制环境变量继承。
   - 限制可导入模块白名单。
   - 限制 stdout/stderr 输出大小。

3. 增加安全审计日志。
   - 记录执行时间、目的、摘要、模式、阻断原因。

4. 明确高风险功能提示与配置入口。
   - CLI、Web Settings、报告中都应说明当前安全模式。

5. 补充敏感操作阻断测试。
   - 覆盖文件读取、网络访问、进程启动、导入高危模块等场景。

#### 验收标准

- `safe` 模式下可稳定阻断文件 IO 与网络访问。
- 高风险执行都能写入审计记录。
- 用户可通过配置明确查看和切换安全模式。

---

### 5.4 模块 D：漏洞验证、证据分级与报告闭环升级

#### 目标

让 VulnClaw 输出的漏洞结论默认更可信，报告默认只代表可证明结果。

#### 需求

1. 拆分 Finding 生命周期。
   - `candidate`
   - `pending_verification`
   - `verified`
   - `rejected`
   - `needs_manual_review`

2. 建立证据等级模型。
   - `L1`：LLM 推断
   - `L2`：路径/差异观察
   - `L3`：工具复现
   - `L4`：独立复核或 PoC 成功

3. 建立验证队列。
   - 候选 finding 不直接等于正式 finding。
   - 持续性任务在每个周期内优先消化高置信度待验证项。

4. 报告输出分层。
   - 主报告只放 `verified`
   - 附录可选展示 `candidate / pending / rejected`
   - 报告中必须展示证据来源、定位信息、复现摘要、验证时间

5. 统一报告生成来源。
   - 会话报告、target-state 报告、周期报告使用同一套 finding render 规则和模板基类。

6. 对误报进行可追踪排除。
   - 被拒绝的 finding 要有 rejection reason，便于后续 resume 与 Web UI 展示。

#### 验收标准

- 正式报告正文不再混入 `pending` 结果。
- 每个 `verified` finding 都包含位置、证据、复现方式和验证时间。
- target-state 恢复时能够基于证据等级优先安排验证队列。

---

### 5.5 模块 E：Target State 治理与恢复能力增强

#### 目标

把 `target_state` 从“好用的持久化功能”升级为“项目的恢复与治理中枢”。

#### 需求

1. 增加 schema 版本字段。
   - 允许后续结构升级时做兼容处理。

2. 提供快照差异能力。
   - 展示新增 finding、状态变化、recon 资产变化、阻塞目标变化。

3. 恢复策略透明化。
   - Web UI 和 CLI 都能展示为什么推荐 `continue_recon / verify_pending_findings / exploit_expand`。

4. 增加恢复上下文压缩规则。
   - 限制 resume summary 长度，同时保留关键证据和失败路径。

5. 对 confidence 计算进行可配置化。
   - observation count、时间衰减、状态权重应可配置，而不是全部硬编码。

6. 提供“只读恢复预览”。
   - 在真正恢复任务前，先展示将采用的 phase、priority targets、blocked targets、next actions。

#### 验收标准

- 任意 target state 均带 schema version。
- CLI 与 Web 均可查看快照差异和恢复策略理由。
- 恢复前用户可预览下一阶段计划，而不是盲目继续。

---

### 5.6 模块 F：Web UI Beta 升级

#### 目标

把当前 Phase 1 数据接线版 UI 升级为单机场景下可真正使用的 Beta 控制台。

#### 需求

1. 完善任务控制台。
   - 支持填写 `max_rounds / rounds_per_cycle / max_cycles / cve / cmd / snapshot_id`
   - 支持选择是否恢复、从哪个快照恢复
   - 支持实时显示当前 phase、工具错误、重试状态、周期摘要

2. 完善 Target State 页面。
   - 展示 verified/pending/rejected 统计
   - 展示恢复策略、priority targets、priority recon assets、blocked targets
   - 展示快照差异，而不仅是原始 JSON

3. 完善 Reports 页面。
   - 支持 Markdown 渲染预览
   - 支持 HTML 直接预览
   - 支持从 target-state 一键生成报告并回看历史版本

4. 完善 Settings 页面。
   - 明确展示 provider、model、安全模式、输出目录、默认轮数
   - 对高风险配置增加说明

5. 增加本地化运行保护。
   - 默认仅监听 `127.0.0.1`
   - 当绑定非本地地址时给出醒目警告

6. 支持任务历史持久化。
   - 当前 `WebTaskManager` 是内存态，刷新页面后历史会丢失
   - Beta 阶段至少应支持最近任务落盘恢复

#### 验收标准

- 用户可通过 Web 完成创建任务、恢复任务、查看事件流、查看 target-state、查看快照、生成报告。
- 刷新页面后最近任务记录不丢失。
- 任务面板不再只展示最新事件文本，而能展示结构化状态。

### 5.7 模块 G：测试、类型、构建与发布工程升级

#### 目标

让项目具备持续演进的工程基础，降低后续改动成本。

#### 需求

1. 修复当前测试漂移。
   - 处理 `tests/test_think_filter.py` 对 `_extract_response` 的旧引用问题。

2. 增加分层测试策略。
   - 单元测试：Agent、Target State、Report、Skill、Config
   - 集成测试：CLI 命令、Web API、任务事件流
   - 合约测试：TaskEvent、TargetView、ReportContentView 等数据模型

3. 引入静态检查门禁。
   - `ruff`
   - `mypy`
   - 可选 `pyright` 或 `pydantic` schema contract 检查

4. 增加 CI 矩阵。
   - Windows
   - Linux
   - Python 3.10 - 3.13

5. 规范发布流程。
   - 构建前版本一致性检查
   - 测试通过后才允许打包
   - 产物与 `__version__`、`pyproject.toml` 一致

6. 补充前端构建检查。
   - `npm build`
   - API 类型与后端 schema 对齐检查

#### 验收标准

- `pytest` 全绿，当前 3 个失败用例被消除。
- CI 至少覆盖 Windows 与 Linux。
- 打包前自动校验版本一致性和前后端构建状态。

## 6. 优先级建议

| 优先级 | 主题 | 说明 |
|------|------|------|
| P0 | Agent 编排统一 | 不先统一内核，后续 Web、报告、恢复都会继续分叉 |
| P0 | 漏洞验证与报告闭环 | 这是产品可信度的核心 |

| P1 | Target State 治理增强 | 这是项目的长期差异化能力 |
| P1 | Web UI Beta 升级 | 当前已投入，值得尽快做成可用版本 |
| P2 | 测试/类型/发布工程 | 对长期维护非常关键，但可以与上面并行推进 |

## 9. 风险与依赖

### 9.1 风险

1. MCP 真接入会带来协议复杂度和环境依赖上升。
2. 强化安全边界可能影响当前某些“灵活但危险”的工作流。
3. finding lifecycle 一旦调整，报告、target-state、Web UI 都要同步改造。
4. Web 任务持久化如果设计不当，容易引入状态一致性问题。

### 9.2 外部依赖

- FastAPI / Uvicorn Web 依赖
- 本地 Node/Vite 构建链
- 未来真实 MCP 服务的运行环境
- Windows 与 Linux 两套 CI 环境

---

## 10. 建议的开发拆分方式

建议按如下 4 条并行主线拆分实施：

1. 内核线
   - Agent 编排统一
   - finding lifecycle
   - target-state schema version

2. 工具线
   - MCP lifecycle
   - 工具错误分类
   - 安全模式与审计

3. 平台线
   - Web API
   - Web Task 持久化
